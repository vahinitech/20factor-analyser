Accuracy & document types

What happens when the page isn't plain prose?

Equations, multiple-choice answers, science notes with diagrams, tiny numbers and symbols all behave differently under the engine. This note explains how the algorithm treats each, how accuracy shifts, and why, plus how Vahini reports the document type it detected.

1What the engine is built for

The Vahini engine measures handwriting geometry: it finds ink, groups it into letters, letters into words, words into lines, then fits a baseline and measures size, spacing, slant, margins and zones. Every one of those steps assumes connected, left-to-right lines of handwritten prose. That is the content type it is calibrated for and where the 20 factors are most accurate.

The short version: the further a page is from "a few lines of normal handwriting," the more the spacing, baseline and slant factors should be read as indicative rather than precise. Vahini detects this and labels the document type and an accuracy expectation directly on the report.

3Why the numbers move

Four concrete mechanisms behind the accuracy shifts above.

Baseline breaks

The engine fits one straight baseline per line. Equations place symbols above and below the line (powers, fractions), so the "baseline" no longer describes a single row, baseline and zone factors lose meaning.

Spacing needs words

Word and letter spacing are computed from the gaps between letters and words. Numbers, symbols and one-word answers don't form normal word groups, so the gap statistics become erratic.

Figures are filtered

To protect the text scores, large connected regions (drawings, boxes, long rules) are identified as figures and removed before measuring. The report says when this happened, the diagram is not assessed.

Resolution sets the floor

Every metric is measured in pixels. Small or blurry writing gives few pixels per stroke, so size, stroke-width and loop-closure measurements get noisier. Sharper, larger photos measure better.

4How Vahini labels the document

Before scoring, a layout classifier inspects the page (it does not read the text) and reports a type and an honest accuracy expectation on the report, on the cover and the sample page.

It reasons from structure only: how many lines and words, average token length, whether large non-text blobs (figures) are present, and ink density. From these it picks one of:

Handwritten prose · High Short answers / few lines · Moderate Numeric / symbolic (equations) · Reduced Text with diagrams / figures · Reduced Sparse / very short · Indicative
Why this matters: a coach or parent can see at a glance whether the score is a confident assessment or a rough read of an unusual page, and the reduced-accuracy types come with a plain-language note explaining what was and wasn't measured.

5Getting the most accurate report

Do

Use the Vahini practice / sample sheet (a few lines of prose). Write at a comfortable size, in good even light, with the phone square to the page.

Avoid for assessment

Equation-heavy maths/physics pages, pages dominated by diagrams, single-word answers, and tiny or blurry photos, fine to explore, but read the scores as indicative.

6Python OpenCV vs JavaScript / TypeScript OpenCV

A common question: are the two "OpenCVs" different? They share the same C++ core and the same maths, the difference is the runtime, coverage and speed, not the correctness of an algorithm.

Python OpenCV (cv2)

Native · server / desktop
  • Thin Python bindings over the full C++ OpenCV library, the complete function set.
  • Fast: SIMD, multi-threading, optional GPU. Best for batch jobs and training.
  • Pairs naturally with PaddleOCR, NumPy, scikit-image, PyTorch.
  • Vahini uses this server-side for heavy recognition and any model training.

OpenCV.js (+ TypeScript)

WebAssembly · in the browser
  • An Emscripten/WASM build of a subset of OpenCV that runs in the browser.
  • Slower than native and a large (~8 MB) download; not every function is exposed.
  • TypeScript adds type definitions only, it's the same opencv.js underneath.
  • Same algorithms → results match within floating-point tolerance; gaps are about coverage, not accuracy.
What Vahini actually does: the in-browser demo engine does not ship OpenCV.js at all. It implements just the primitives it needs, Otsu & adaptive thresholding, connected components, baseline regression, gradient-orientation slant, in plain JavaScript. That keeps the demo tiny, instant and fully offline, with no 8 MB WASM download. The maths is identical to OpenCV's, so the measurements agree; the production server can use Python OpenCV + PaddleOCR for the heavy lifting. Same algorithms, two runtimes, chosen per job.
